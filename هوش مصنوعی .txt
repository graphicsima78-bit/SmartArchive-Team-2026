1) قانون «فایل همنام کنار فایل اصلی» (Sidecar Pairing)
برای هر فایل گرافیکی (مثل .psd, .ai, .cdr, .blend, .fig, …) قبل از انتقال/کپی، سیستم این کار را می‌کند:
1) اسم پایه را درمی‌آورد:
مثال: poster_v3.psd → پایه: poster_v3
2) داخل همان پوشهٔ مبدا دنبال فایل‌های “هم‌پایه” می‌گردد، مثل:
poster_v3.jpg
poster_v3.png
poster_v3.webp
poster_v3.jpeg
3) اگر پیدا کرد:
همراه فایل اصلی به همان مسیر مقصد منتقل/کپی می‌شود
و اگر در مقصد اسم تکراری بود، با suffix مثل _copy1 ذخیره می‌شود
این باعث می‌شود هرجا فایل گرافیکی دسته‌بندی شد، عکس نمایشی‌اش هم همان‌جا باشد.

دسته بندی کلی اولی باشه و بعد بقیه بعد از کپی فایل در حال کار و بعد قطع شود زبان پیش فرض انگلیسی و لی این ربطی به قسمت آهنگ که میخواد اسم خواننده رو فارسی یا انگلیسی بزنه نداره داخل اون قسمت بایت یک قسمت داشته باشه بکه بتونی دستی اون رو انتخاب کنی که به کدوم صورت دسته بندی کنه

تصمیم‌های UI نهایی (طبق پیام شما)
1) Tab اول = «دسته‌بندی کلی» (همان انتقال ساده/بدون تغییر) و اولین تب باشد.  
2) دکمه Stop این‌طور عمل کند:  
بعد از تمام شدن کپیِ فایلِ در حال پردازش، قطع کند (Safe stop).  
3) زبان پیش‌فرض UI = English.  
4) بخش موسیقی مستقل از زبان UI باشد و داخل همان تب، یک گزینه دستی داشته باشد که نام خواننده‌ها در پوشه‌بندی فارسی باشد یا انگلیسی.
دقیقاً چه گزینه‌ای برای موسیقی اضافه می‌کنیم؟
در Tab «Audio/Music» یک انتخاب اضافه می‌شود:
Artist naming / Folder language
( ) Persian (فارسی)
( ) English (Original/Latin)
رفتار:
اگر Persian انتخاب شد:
اگر خواننده در نگاشت ایرانی‌ها شناخته شد → پوشه با نام فارسی استاندارد ساخته می‌شود (فقط یک پوشه).
اگر شناخته نشد → نام اصلی، تمیز و یکدست می‌شود.
اگر English انتخاب شد:
حتی اگر نگاشت فارسی وجود داشت، پوشه با نام انگلیسی/اصلی ساخته می‌شود (باز هم یکدست و یکی).
اسم فایل آهنگ هم طبق انتخاب شما:
trackNo - Artist - Title.ext
اگر trackNo نبود: Artist - Title.ext
رفتار Stop/Pause دقیقاً چطور پیاده می‌شود؟
Stop (Safe): یک فلگ “stop requested” فعال می‌شود، برنامه:
اجازه می‌دهد همان فایل جاری کامل کپی شود
بعد بلافاصله حلقه را متوقف می‌کند
Pause/Resume: بین فایل‌ها (و در کپی‌های بزرگ در chunkها) مکث می‌کند تا دقیق و بدون خراب شدن فایل‌ها باشد.
اسم تب اول هم نهایی شد
Tab 1 title:
فارسی: «دسته‌بندی کلی»
انگلیسی: General (No changes) یا General Transfer
(چون UI پیش‌فرض انگلیسی است، عنوان انگلیسی را پیش‌فرض می‌گذاریم و با تغییر زبان، فارسی می‌شود.)
اگر تایید می‌کنی، من از اینجا به بعد Spec را “قفل‌شده” در نظر می‌گیرم:
ترتیب تب‌ها: General اول، بعد Images، Graphics/Vectors، Audio، Software
Stop = بعد از فایل جاری
UI default = English
Music دارای انتخاب Persian/English برای خواننده‌ها (مستقل از UI language)
فقط بالای این قسمت نوشته بشه که ایا میخواهید نام خواننده و آهنگ به فارسی باشد یا انگلیسی بعد زیر آن تیکها باشند که هر کدام انتخاب شد همان مورد انجام شود

باشه—دو مورد رو نهایی می‌کنم و بعد درباره GitHub شفاف می‌گم چه کاری از دستم برمیاد:
1) تنظیم پیش‌فرض زبان نام‌ها در موسیقی
در تب Music:
پیش‌فرض: English
یک تیک/گزینه: Persian
اگر تیک Persian روشن شد → نام خواننده و آهنگ (پوشه و فایل) فارسی‌سازی/فارسی‌نویسی می‌شود
اگر خاموش بود → انگلیسی می‌ماند
متن بالای گزینه‌ها هم دقیقاً همونی که گفتی:
«آیا می‌خواهید نام خواننده و آهنگ به فارسی باشد یا انگلیسی؟»
2) تمیز کردن و جایگزین کردن پروژه داخل GitHub
من از داخل این محیط دسترسی نوشتن به ریپوی GitHub شما ندارم (فقط می‌تونم بخونم/تحلیل کنم). یعنی نمی‌تونم مستقیم:
فایل‌ها رو حذف کنم
ساختار رو عوض کنم
commit/push انجام بدم
اما می‌تونم دقیقاً همون نتیجه‌ای که می‌خوای رو به دو شکل بهت بدم:
گزینه A (سریع و تمیز): من ساختار نهایی پروژه را “فایل به فایل” بهت می‌دم، تو فقط جایگزین می‌کنی
من یک ساختار جدید و یکدست پیشنهاد می‌دم (مثلاً):
app/ (UI)
engine/ (پردازش و دسته‌بندی)
rules/ (taxonomy ها)
i18n/ (fa/en)
assets/ (آیکون‌ها)
requirements.txt
.gitignore
README.md
بعد تو:
1) یک branch جدید بساز (مثلاً refactor-v2)
2) فایل‌های قبلی اضافی/zipها و پوشه‌های قدیمی را حذف کن
3) این ساختار جدید را کپی کن
4) commit/push
گزینه B: من دقیقاً لیست “چی حذف شود / چی بماند” می‌دم
برای ریپوی فعلی شما، مواردی که معمولاً باید حذف/خارج شوند:
فایل‌های zip بزرگ (مثل 69.zip, 70.zip, 71.zip و …) → بهتر در GitHub Releases باشند، نه داخل repo
نسخه‌های تکراری و فولدرهای legacy که فقط حجم می‌سازن
خروجی‌های build/dist (اگر هست)
و در عوض، فقط سورس + تنظیمات + راهنما باقی بماند.
فقط یک سؤال خیلی کوتاه برای اینکه دقیقاً ریپو را “جایگزین” کنیم
می‌خوای ریپو از همینجا تبدیل بشه به نسخه جدید (V2) و قدیمی‌ها برن کنار، یا می‌خوای قدیمی‌ها در یک پوشه legacy/ نگه داشته بشن؟
V2 فقط (تمیزترین حالت): همه چیز جدید، legacy حذف
V2 + legacy: قدیمی‌ها منتقل به legacy/ (برای آرشیو)
یکی رو بگو؛ بعدش من:
ساختار فولدر نهایی
لیست فایل‌های لازم
.gitignore دقیق
و متن README
را کامل و آماده می‌دم تا دقیقاً داخل GitHub Desktop جایگزین و push کنی.
